# Release v0.25.0 — Owner-Qualified Feature IDs + Equipment ID Normalization

**Release Date:** June 17, 2026
**Package Version:** 0.25.0
**Breaking Changes:** Yes — feature IDs and equipment IDs change shape. See *Migration*.

## Overview

v0.24.0 landed a data-quality audit and closed every cross-reference
error. The two remaining classes of audit findings — duplicate feature
IDs across classes and hyphenated equipment IDs — required breaking the
consumer-facing ID contract, which is what v0.25.0 does. After this
release the audit reports **zero** `bad_id_format` and **zero**
`duplicate_id` findings on the `features` and `equipment` datasets
(22 pre-existing duplicates in `rules` and `tables` remain, tracked
for v0.26.0).

The bundle is intended to go to Blackmoor for external review next, so
both ID changes are landed together in a single breaking release rather
than split across two.

Cross-reference validation: **still 0 errors**.

## What changed

### Phase D — Feature IDs disambiguated by owner

Feature IDs now embed the owning class or lineage:

| Before | After |
|---|---|
| `feature:rage` | `feature:barbarian:rage` |
| `feature:ability_score_improvement` (×12 collisions) | `feature:barbarian:ability_score_improvement`, `feature:bard:…`, … |
| `feature:darkvision` (×4 collisions) | `feature:dwarf:darkvision`, `feature:elf:darkvision`, `feature:gnome:darkvision`, `feature:tiefling:darkvision` |
| `feature:spellcasting` (×7 collisions) | `feature:bard:spellcasting`, `feature:cleric:…`, `feature:druid:…`, … |

A new required `owner_id` field accompanies every feature record so
consumers don't need to parse the id to find the owner:

```json
{
  "id": "feature:barbarian:rage",
  "name": "Rage",
  "simple_name": "rage",
  "owner_id": "class:barbarian",
  "page": 8,
  ...
}
```

`parse_classes._build_class_record` rewrites each class's `features`
array to the new qualified form via `_qualify_feature_ids()`, so
`class:barbarian.features[0] == "feature:barbarian:rage"`.

#### How owners are resolved

`parse_features._resolve_owners` walks the extracted features in PDF
order and stream-matches them against the canonical CLASS_DATA /
LINEAGE_DATA feature sequences (which mirror the PDF reading order).
Universal per-class headings absent from any single class's curated
list (e.g., the per-class `Ability Score Improvement` reprints) are
pinned to the current owner so they don't corrupt the cursor.

Result: **245 unique feature IDs** spread across all 12 classes and
all 13 lineages, with zero duplicates.

### Phase E — Equipment IDs normalized to underscores

`parse_equipment._generate_id` and the assemble-equipment pack-id
generator both route through `postprocess.ids.normalize_id`, so:

| Before | After |
|---|---|
| `item:battle-axe` | `item:battle_axe` |
| `item:rations-1-day` | `item:rations_1_day` |
| `item:string-10-feet` | `item:string_10_feet` |

The hardcoded item IDs in `equipment_extended.py` and
`equipment_packs.py` (and the pack-content references that point at
them) are rewritten in-place to the underscore form. Hyphenated
`simple_name` values in `equipment_extended.py` are normalized to
match the new ID convention.

The audit's `ID_RE` is also broadened to accept multi-segment IDs
(`^[a-z][a-z0-9_]*(:[a-z0-9_]+)+$`) so the new feature ID shape
passes format validation.

### Schema version bumps

| Schema | Old | New | Rationale |
|---|---|---|---|
| `features.schema.json` | 2.0.0 | **3.0.0** | Major: `id` pattern tightened to require 2 segments; new required field `owner_id`. Existing v0.24.0 documents do not validate. |
| `class.schema.json` | 2.0.0 | **2.1.0** | Minor: only the `features[]` description changed; no validator-level constraint changed. |
| `equipment.schema.json` | 2.0.0 | **2.1.0** | Minor: `id` pattern tightened from `[a-z0-9_-]+` to `[a-z0-9_]+`. All v0.25.0 data conforms; a stricter validator on prior data would have failed. |

## Migration

Every consumer that hardcodes a feature ID needs to update. The new
shape is mechanical to derive: prepend the owner's `simple_name` after
the `feature:` prefix.

```python
# Before
"feature:rage"
# After
"feature:barbarian:rage"
```

For equipment IDs, do a global hyphen → underscore replacement on
anything after `item:`:

```python
# Before
"item:battle-axe"
# After
"item:battle_axe"
```

If you read features via cross-reference from a class record, no
code change is required — `class.features` already contains the
new IDs.

If your code parsed feature ownership from the ID (`id.split(":")[1]`),
switch to reading the new `owner_id` field instead.
