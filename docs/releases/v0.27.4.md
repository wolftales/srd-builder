# v0.27.4 — Equipment page-constants drift fixed + cross-extractor audit

## TL;DR

`extract_equipment.py` was silently dropping PDF page 74 on every
build. v0.27.4 fixes the off-by-one, harmonizes the page-constant
convention across all four per-dataset extractors, and pins them to
the canonical `PAGE_INDEX` table via a new parametrized audit test
so future drift fails CI.

- **8 rows** recovered from PDF page 74 (Bread loaf, Inn-stay tiers,
  Skilled hireling, Messenger, Ship's passage, etc.).
- **4 new test assertions** (`464 passed`, up from `460`).
- **No fixture changes** — the bug only affected the live PDF
  extractor; fixture-driven golden tests were unaffected.

## What was wrong

Three of the four per-dataset extractors used a 1-indexed PDF page
convention (constants store the actual SRD page label; iteration does
`range(start - 1, end)`):

| Extractor | Constants | PAGE_INDEX | Match? |
| --- | --- | --- | --- |
| `extract_spells.py` | `114`–`194` | `114`–`194` | ✓ |
| `extract_magic_items.py` | `206`–`253` | `206`–`253` | ✓ |
| `extract_conditions.py` | `358`–`359` | `358`–`359` | ✓ |
| `extract_equipment.py` | `61`–`72` (0-indexed) | `62`–`74` | ✗ |

`extract_equipment.py` was the outlier: 0-indexed *and* off by one on
the end constant. Its iteration was `range(start, end + 1)` covering
PyMuPDF indexes 61–72 = PDF pages 62–73 — silently skipping PDF page 74
(Services / Lifestyle / Spellcasting Services tables) on every build.

The bug was invisible because the dropped page's content was
hand-curated in
[`equipment_descriptions.py`](../../src/srd_builder/assemble/equipment_descriptions.py)
under `LIFESTYLE_DESCRIPTIONS` (see the comment on its line 352:
*"Lifestyle descriptions from SRD 5.1 page 73"*). That hand-curated
data is the workaround that masked the drift.

## What changed

### `src/srd_builder/extract/datasets/extract_equipment.py`

Constants and iteration converted to 1-indexed:

```python
# Before
EQUIPMENT_START_PAGE = 61  # Page 62
EQUIPMENT_END_PAGE = 72  # Page 73
for page_num in range(EQUIPMENT_START_PAGE, EQUIPMENT_END_PAGE + 1):
    page = doc[page_num]

# After
EQUIPMENT_START_PAGE = 62  # PDF page 62 (chapter title "Equipment")
EQUIPMENT_END_PAGE = 74    # PDF page 74 (Services / Lifestyle / Spellcasting Services)
for page_num in range(EQUIPMENT_START_PAGE - 1, EQUIPMENT_END_PAGE):
    page = doc[page_num]
```

Now matches the iteration convention used by `extract_spells.py`,
`extract_magic_items.py`, and `extract_conditions.py`. `pages_processed`
in the extractor `_meta` block now correctly reports `[62, 63, …, 74]`.

### New audit test: `tests/test_pdf_provenance.py`

```python
@pytest.mark.parametrize(
    ("module_path", "start_attr", "end_attr", "page_index_key"),
    [
        ("srd_builder.extract.datasets.extract_equipment",
         "EQUIPMENT_START_PAGE", "EQUIPMENT_END_PAGE", "equipment"),
        ("srd_builder.extract.datasets.extract_spells",
         "SPELL_START_PAGE", "SPELL_END_PAGE", "spell_descriptions"),
        ("srd_builder.extract.datasets.extract_magic_items",
         "MAGIC_ITEMS_START_PAGE", "MAGIC_ITEMS_END_PAGE", "magic_items"),
        ("srd_builder.extract.datasets.extract_conditions",
         "CONDITION_START_PAGE", "CONDITION_END_PAGE", "appendix_ph_a_conditions"),
    ],
)
def test_extractor_page_constants_agree_with_page_index(...): ...
```

The test asserts each extractor's `(START, END)` constants equal
`PAGE_INDEX[key]["pages"]`. If anyone re-introduces a 0-indexed
constant or trims a page by accident, this fails before the build
silently drops content.

## What did NOT change

- **No fixture regeneration.** Golden tests
  (`test_golden_equipment.py`, etc.) use captured raw-extractor output
  fixtures, not the live PDF. They were unaffected.
- **No `equipment_descriptions.py` retirement.** The hand-curated
  `LIFESTYLE_DESCRIPTIONS` block is left in place pending the broader
  descriptions retirement. The bug fix recovers the tabular rows
  (item name + cost) for page 74, but the prose descriptions remain
  hand-curated for now.
- **No `Document.get_toc()` lookup.** The originally-planned approach
  (replace hardcoded constants with PDF TOC lookup) was investigated
  and abandoned: the SRD PDF's TOC only exposes 2 file-level entries
  (`SRD-CC_V1.1 preamble.pdf` p.1, `SRD-OGL_V5.1.pdf` p.3) with no
  section anchors. `PAGE_INDEX` (the existing canonical table in
  `src/srd_builder/utils/page_index.py`) is the source of truth instead.
