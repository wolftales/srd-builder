# Release v0.27.1 — Prose Section Splitter Fix + Poison Probe

**Release Date:** June 18, 2026
**Package Version:** 0.27.1
**Breaking Changes:** none

## Overview

v0.27.1 is a focused bug-fix release. It picks up the
`poison_descriptions.py` reproducer that landed after the v0.27.0 tag
(the fourth "PDF text is corrupted" claim to fail under scrutiny), and
then fixes the actual upstream bugs that had been blocking the live
poison-description extractor for months.

The pattern is the same as the lineage / spell-class / class probes
shipped in v0.27.0: a hand-curated rationale written years ago turns
out to be wrong, the text is fully extractable, and the real issue is
a small parser bug — not a PDF problem.

The fixes also surface a quietly-broken whitespace cleaner that was
never doing what its comment said: `clean_text`'s smart-quote
substitution had been source-mangled into a no-op years ago and
silently leaked U+2019 / U+201C / U+201D through every dataset that
flowed through it.

## What changed

### Poison descriptions reproducer (initially DISPUTED, now fixed)

Added [`tests/test_pdf_provenance.py::test_poison_pages_are_extractable_after_whitespace_normalization`](../../tests/test_pdf_provenance.py)
asserting that all 14 SRD poison names, all 5 DC values, and the
standard mechanic keywords ("Constitution saving throw", "poison
damage", duration phrases) appear in pages 204–205 after the standard
whitespace normalization. They do — every one.

This was the fourth claimed-corrupted region in the codebase. Like
lineages, spell_classes, and class_targets before it: same era, same
author, same "manually transcribed" rationale, same outcome under
scrutiny.

### `clean_text` smart-quote substitution (was a no-op)

[src/srd_builder/postprocess/text.py](../../src/srd_builder/postprocess/text.py)
contained the line:

```python
text = text.replace("'", "'")  # smart quotes
```

Both arguments are ASCII `U+0027`. The substitution did nothing. The
adjacent double-quote substitutions had been mangled the same way
(`text.replace(""", '"')` — both ends literal). The intent was
clearly to fold curly quotes to ASCII, but the file had at some point
been round-tripped through a smart-quote system that ate the
intended Unicode characters.

Fix: rewrite all five substitutions with explicit Unicode escapes
(`\u2018`, `\u2019`, `\u201b`, `\u201c`, `\u201d`) so they cannot
regress again.

Visible downstream effects (now baked into fixtures):

| Dataset | U+2019 normalized | Notes |
| --- | --- | --- |
| conditions, rules | yes | text flows fully through `clean_text` |
| monsters | 497 of 498 | one curly apostrophe in a name field bypasses `clean_text` — left for a follow-up parser pass |
| features, magic_items, tables | unchanged | these datasets don't pipe through `clean_text` today |

### `split_by_known_headers` no longer locks onto preamble headers

[src/srd_builder/utils/prose.py](../../src/srd_builder/utils/prose.py)
matched the *first* occurrence of each header in the cleaned text. For
prose sections that have a price / summary table at the top of the
page repeating header words, this trapped the splitter on the table
row. The poison pages were the canonical case:

- Pages 204–205 open with a price table (`Malice Inhaled 250 gp`,
  `Torpor Ingested 600 gp`, …).
- The splitter found `Malice` at the table row, then over-extended the
  section all the way to the next body header — `malice` absorbed
  ~2000 chars from neighbouring sections, `torpor` ~3700.

Fix: `split_by_known_headers` now accepts an optional
`start_marker=` keyword. When provided, the per-header search begins
only after the marker's first occurrence. `ProseExtractor` and
`assemble_prose._extract_raw_sections` thread the parameter through
from `TABLES` config. The poison_descriptions config sets
`start_marker="Sample Poisons"` and re-includes `"Assassin's Blood"`
in `known_headers` (it had been excluded entirely with a stale
"not extractable" comment).

### Manual `POISON_DESCRIPTIONS` dead-key bug

[src/srd_builder/rulesets/srd_5_1/poison_descriptions.py](../../src/srd_builder/rulesets/srd_5_1/poison_descriptions.py)
keyed the assassin's blood entry as `"assassin's_blood"` with a U+2019
fancy apostrophe and a comment proudly noting it. But the lookup site
goes through `normalize_id("Assassin's blood")` which strips
apostrophes entirely and produces `"assassins_blood"`. The key had
never matched and the entry had been silently inert in production.

Fix: renamed the key to `"assassins_blood"`. The assassin's-blood
poison record in [tests/fixtures/srd_5_1/normalized/poisons.json](../../tests/fixtures/srd_5_1/normalized/poisons.json)
now carries the description / save / damage fields it should always
have had.

### End-to-end regression test

Added [`tests/test_pdf_provenance.py::test_poison_descriptions_extract_all_14_sections_cleanly`](../../tests/test_pdf_provenance.py)
asserting that:

- all 14 expected `simple_name` values come out of the live extractor;
- no description exceeds 800 chars (pre-fix malice was ~2000, torpor
  ~3700);
- every entry has a parsed Constitution save DC;
- the extractor produces zero warnings.

## Retirement status — unchanged

`poison_descriptions.py` stays in the tree. The live extractor now
produces clean 14/14 sections, but the parser does not yet recognize
the "taking X poison damage on a failed save" damage-formula phrasing
used by `midnight_tears`, `purple_worm_poison`, `serpent_venom`, and
`wyvern_poison`. `parse_poisons_table.py` prefers `POISON_DESCRIPTIONS`
and falls back to the live extractor; full retirement is gated on a
parser pass, not on extraction.

  `monsters.json` is in a name field that bypasses `clean_text`.
