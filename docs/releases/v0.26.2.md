# Release v0.26.2 — Structural Cleanup (finishing attempt #2)

**Release Date:** June 17, 2026
**Package Version:** 0.26.2
**Breaking Changes:** `source` field value changed from `"SRD 5.1"` to canonical `"SRD_CC_v5.1"` in every dataset record. Downstream consumers that filter on `source` must update the literal.
**Migration:** Update any `source == "SRD 5.1"` checks to `source == "SRD_CC_v5.1"` (or, preferably, to a `startswith("SRD_CC_v5.1")` check so `"SRD_CC_v5.1 (extended)"` from equipment is also matched).

## Overview

v0.26.2 is a **structural cleanup** release. No new datasets, no new schemas,
no new parsers — just finishing the work that previous "attempt #2"
reorganizations left half-done. The repo now has:

- one extraction directory (`extract/`), not two (`extract/` + `extraction/`)
- one home for hand-curated per-ruleset Python data (`rulesets/srd_5_1/`),
  not three (`srd_5_1/`, `data/`, `extraction/`)
- one source of truth for ruleset identity (`RULESETS` registry in
  `constants.py`), not a lone `DATA_SOURCE` constant + scattered string literals
- one shared prose-helper module (`utils/prose.py`), not three near-duplicate
  inline implementations
- one shared source-stamping helper (`utils.metadata.stamp_source`), not
  ad-hoc `[{**r, "source": "SRD 5.1"} for r in records]` repeated in 14 files

The canonical source identifier `"SRD_CC_v5.1"` now matches the PDF filename
(`SRD_CC_v5.1.pdf`) and the registry id — closing a small but persistent
naming gap between the data-on-disk and the data-in-records.

## What changed

### Phase A — Module relocations

- `validate_references.py` moved from `src/srd_builder/` to `src/srd_builder/utils/`
  (it's a utility, not a top-level entry point).
- `table_targets.py` moved from `scripts/` to `src/srd_builder/extract/`
  (it's runtime config consumed by the extractor, not a script).

### Phase B — Duplication audit

Investigated several "looks like a duplicate" signals across `parse/` and
`postprocess/`. None warranted consolidation (the apparent duplication was
schema-shaped, not code-shaped). Recorded for future reference; no code change.

### Phase C — Per-ruleset Python data consolidation

`class_targets.py`, `lineage_targets.py`, `spell_class_targets.py`, and
`poison_descriptions.py` consolidated under
`src/srd_builder/rulesets/srd_5_1/` alongside the existing PDF and `raw/` dirs.
Previous homes (`srd_5_1/`, `data/`, scattered) are gone.

### Phase D — `extraction/` → `extract/`

The two-pipeline-layer naming inconsistency is resolved:

- The table-extraction engine (`extractor.py`, `patterns.py`,
  `extraction_metadata.py`, `text_parser_utils.py`, `table_targets.py`) now
  lives at the top of `extract/`.
- The bespoke per-dataset extractors (`extract_<dataset>.py`) are grouped
  under `extract/datasets/`.
- All imports updated. `extraction/` no longer exists.

### Phase D prep — `utils/prose.py`

Three near-duplicate prose-helper implementations (split-on-headings,
paragraph normalization, the usual) consolidated into one
`src/srd_builder/utils/prose.py` consumed by `assemble_prose.py` and the
prose-shaped extractors.

### Phase E1 — Workspace dirname literals

`"raw"`, `"dist"`, `"rulesets"`, `"extracted"`, etc. — the strings that name
on-disk subdirectories — swept into named constants in `constants.py`. The
literals were duplicated across ~30 call sites; now they live in one place.

### Phase E2.1 — `RULESETS` registry

Replaced the single `DATA_SOURCE = "SRD 5.1"` constant with:

```python
RULESETS: Final[dict[str, dict[str, str]]] = {
    "srd_5_1": {
        "id": "srd_5_1",
        "source_id": "SRD_CC_v5.1",
        "display_name": "D&D 5e (2014 SRD)",
        "edition_short": "5e",
        "ruleset_version": "5.1",
        "pdf_filename": "SRD_CC_v5.1.pdf",
    },
}
```

This is the foundation for future multi-ruleset support (SRD 5.2, OGL 1.0a,
homebrew rulesets); v0.26.2 only ships the `srd_5_1` entry.

### Phase E2.2 — Ruleset threading + source normalization

Every parser, assembler, and extractor that stamps `"source"` now takes a
`ruleset` parameter and resolves the source identifier from the `RULESETS`
registry. Hardcoded `"SRD 5.1"` literals are gone (except in two
human-readable WotC attribution strings in `utils/metadata.py` and
`build.py`, which are intentionally not registry-derived).

- New helper: `utils.metadata.stamp_source(records, ruleset)` —
  centralizes the `[{**rec, "source": source} for rec in records]` pattern
  used by static-list parsers.
- Source value normalized: `"SRD 5.1"` → `"SRD_CC_v5.1"` in every record
  output and every test fixture. Equipment retains its
  `"SRD_CC_v5.1 (extended)"` suffix for items added by `equipment_extended.py`.
- Parser-stamped values are now trusted in postprocess: `clean_rule_record`
  no longer applies a `"SRD 5.1"` fallback.

### Phase F prep — `bump_version.py`

The fixture-regeneration helper was passing `"SRD 5.1"` literals in
hand-built record dicts (classes, tables) and was missing `ruleset` kwargs
on the new parser signatures. Updated end-to-end:

- All `parse_*` lambdas in `fixtures` pass `"srd_5_1"`.
- Hand-built class/table record literals use `"SRD_CC_v5.1"`.
- `reference_fixtures` loop calls `parse_fn("srd_5_1")` for
  `ability_scores` / `damage_types` / `skills` / `weapon_properties`.
- `features` excluded from auto-regen (regeneration needs source-type
  partitioning for `owner_id` resolution that the raw fixture lacks;
  documented in script comments — regenerate via full `make output`
  + copy from `dist/features.json`).
- Drive-by: also threads the `ruleset` parameter through every fixture
  helper consistently, so future bumps work the first time.

### Drive-by — PEP 758 cleanup

Python 3.14 / PEP 758 allows unparenthesized multi-type `except` clauses
(`except OSError, json.JSONDecodeError:`). Linters and editor tooling
don't yet flag these consistently and they read like Python-2 tuple-syntax
errors. Four sites normalized to the parenthesized form:

- `src/srd_builder/utils/metadata.py:162`
- `src/srd_builder/parse/parse_actions.py:105, 156`
- `src/srd_builder/postprocess/monsters.py:227`

### Drive-by — `tests/test_equipment_packs.py`

Fixed a stale `from src.srd_builder...` import to use the canonical
`from srd_builder...` form (the `src.` prefix only worked when the
project root was on `sys.path`, which is true under `python -m pytest`
but not under bare `pytest tests/` as `bump_version.py` invokes).

## Bundle changes

`source` field value in every dataset record changes from `"SRD 5.1"` to
`"SRD_CC_v5.1"`. Otherwise no schema, layout, or count changes — bundle
shape is identical to v0.26.1.

## Tests

- v0.26.1 baseline: **294 passed, 19 skipped**
- v0.26.2: **294 passed, 19 skipped** — no test count change, all fixtures
  regenerated and matching.

## Schema versions

No schema versions bumped in this release. The `source` field value
change is data-level, not schema-level (the schemas already constrain
`source` only to "string"). The `RULESETS` registry pinning could
motivate a future minor bump of the four reference-dataset schemas to
add a `source` enum constraint — captured in the parking lot for v0.27.0.
