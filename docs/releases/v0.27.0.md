# Release v0.27.0 — Hand-Curated Data Retirement

**Release Date:** June 17, 2026
**Package Version:** 0.27.0
**Breaking Changes:** `srd_builder.postprocess.spells.clean_spell_record()`
  now takes an explicit `spell_classes_map=` keyword argument. The previous
  implicit lookup against `rulesets.srd_5_1.spell_class_targets.get_spell_classes()`
  is gone (the module is gone). Callers that need the `classes` field
  attached must build the map via
  `srd_builder.extract.datasets.extract_spell_classes.build_spell_to_classes_map()`
  (live extractor) or load the snapshot fixture at
  `tests/fixtures/srd_5_1/spell_class_targets_snapshot.json`. Calls that
  omit the keyword silently skip the `classes` field — this is a deliberate
  fail-loud-at-the-call-site rather than fail-silent-with-stale-module choice.
**Migration:**
  - Build pipeline (`build.py`): already updated; live extractor wired in.
  - Golden test (`tests/test_golden_spells.py`): already updated; loads snapshot.
  - Fixture refresher (`scripts/bump_version.py`): already updated; loads snapshot.
  - Any external code calling `clean_spell_record(spell)` directly must add
    `spell_classes_map=…` if it wants the `classes` field populated.

## Overview

v0.27.0 retires the largest hand-curated data surfaces in the project —
files marked "PDF text is corrupted" or "manually transcribed via visual
inspection" — by replacing them with live extractors that walk the SRD 5.1
PDF directly. Both rationales were disproved by reproducer tests under
`pymupdf 1.27.x` once whitespace was normalized.

Net result:

- **~1,244 lines** of hand-curated data deleted (`lineage_targets.py`
  + `spell_class_targets.py`, plus the dead `scripts/validate_spell_classes.py`).
- **2 new live extractors** built on `utils.pdf_probe` and
  `utils.page_index` (the disciplined extraction substrate AGENTS.md
  mandates).
- **1 new reproducer test** for the third claimed-corrupted region
  (`class_targets.py` pp. 8–55) — text is fully extractable; full
  retirement deferred because the structural payload is much richer
  than a flat spell list.
- **62 new test assertions** total across the new extractor tests and
  reproducer probe.

This release closes the "PDF corruption" lore that quietly accumulated
~2,000 lines of hand-curated data over previous releases. All three
"PDF text is corrupted" claims in the codebase — same era, same author —
have now had reproducers written. All three failed. The text was
always extractable; the parser was just missing the standard
`\t\r\xa0` whitespace normalization.

## What changed

### P1 — Retire `lineage_targets.py` (326 lines)

The hand-curated `LINEAGE_DATA` list (9 base lineages + 4 subraces, with
ability modifiers, size, speed, traits, languages, and page numbers) is
replaced by [src/srd_builder/extract/datasets/extract_lineages.py](../../src/srd_builder/extract/datasets/extract_lineages.py).

Detection rules (font-fingerprint walk over PDF pages 3–7):

- 18pt GillSans-SemiBold = race heading
- 12pt G-SB after first race = subrace heading
- 9.8pt Cambria-BoldItalic ending in `.` = trait header

Special parsers handle the irregular ability-modifier prose:

- Human "each +1" expands to all six abilities.
- Half-Elf "two other ability scores" emits `"other": 2` plus an
  `ability_modifier_note` field for downstream consumers.

One deliberate PDF-fidelity correction: the legacy data called the
Halfling subrace "Lightfoot Halfling" but the PDF heading literally
reads just "Lightfoot". The extractor uses the PDF spelling; the test
pins this via a `_SUBRACE_NAME_OVERRIDES` map in
[tests/test_extract_lineages.py](../../tests/test_extract_lineages.py).

Tests: 49 assertions across 9 races + 4 subraces.

### P2 — Retire `spell_class_targets.py` (917 lines)

The hand-curated 8-class `*_SPELLS` lists (the largest hand-curated
surface in the project) are replaced by
[src/srd_builder/extract/datasets/extract_spell_classes.py](../../src/srd_builder/extract/datasets/extract_spell_classes.py).

Detection rules (font-fingerprint walk over PDF pages 105–113):

- 13.9pt GillSans-SemiBold = class header
  (`Bard Spells`, `Cleric Spells`, …, `Wizard Spells`)
- 12pt G-SB = spell-level header (skipped)
- 9.8pt Cambria = spell name (`normalize_id` applied to produce
  `simple_name`)
- 18pt G-SB "Spell Lists" page heading and 10.5–11.0pt page-number /
  footer runs are skipped.

The extractor produces a **byte-perfect 778-for-778 match** against the
retired `*_SPELLS` lists across all 8 caster classes — zero divergence.

Consumer refactor: `clean_spell_record()` now takes an explicit
`spell_classes_map=` keyword (default `None` → no `classes` field
attached) instead of importing a module-level lookup. The `build.py`
pipeline wires the live extractor; the golden test and the fixture
refresher load a deterministic snapshot fixture
([tests/fixtures/srd_5_1/spell_class_targets_snapshot.json](../../tests/fixtures/srd_5_1/spell_class_targets_snapshot.json))
so neither needs to re-run the live PDF walk on every fixture refresh.

Tests: 12 assertions (per-class set equality against snapshot +
multi-class spell spot-checks: `fireball` → `[sorcerer, wizard]`,
`cure_wounds` ⊇ `{bard, cleric, druid, paladin, ranger}`,
`eldritch_blast` → `[warlock]`).

### P3 — Probe `class_targets.py` (763 lines, DISPUTED, not retired)

Added [tests/test_pdf_provenance.py::test_class_pages_are_extractable_after_whitespace_normalization](../../tests/test_pdf_provenance.py)
asserting that all 12 class names, 15 sampled well-known features
(Rage, Spellcasting, Sneak Attack, Action Surge, Wild Shape, Lay on
Hands, Divine Sense, Favored Enemy, Cunning Action, Sorcerous Origin,
Pact Magic, Arcane Recovery, Bardic Inspiration, Channel Divinity),
and standard section headers (Hit Points, Proficiencies, Equipment,
Primal Path, Bard College, Divine Domain) appear in pages 8–55 after
the standard whitespace normalization.

**All 33 keywords found.** The "manually transcribed via visual
inspection" rationale is the same era / same author as the lineage and
spell-class claims already disproved.

`class_targets.py` is **not retired in v0.27.0** — its structural
payload (proficiencies dict with armor/weapons/tools/skills, feature-ID
lists, subclass mapping, 20-level progression) is significantly richer
than a flat spell list, so a real `extract_classes.py` will need a
larger font-fingerprint + table walk. `docs/PROVENANCE.md` now marks
the entry **DISPUTED** with the reproducer linked, deferring retirement
to a future release.

## Lessons captured

The pattern across all three retirements is identical and worth
codifying in AGENTS.md style for future contributors:

1. **Don't trust "PDF corruption" claims at face value.** Three for
   three in this codebase have failed under reproducer scrutiny once
   `\t\r\xa0` whitespace normalization is applied. The legacy claims
   were written years ago against an older PyMuPDF.
2. **Snapshot the hand-curated data before deleting it.** Both
   retirements went through the same sequence:
   write extractor → diff against legacy lists → snapshot the legacy
   data to a JSON fixture → cut callers over → delete the module. The
   snapshot becomes the deterministic test reference, decoupling
   fixture-refresh runs from live PDF walks.
3. **Prefer PDF fidelity over legacy embellishments.** The "Lightfoot
   Halfling" → "Lightfoot" change was an opportunity to surface a
   silent legacy edit. Pin such decisions in the test, never the
   extractor.
