# Release v0.24.0 — Data Quality Audit + Footer/ID Hardening

**Release Date:** June 17, 2026
**Package Version:** 0.24.0
**Breaking Changes:** No (additive fixes; `meta.json` shape changed but no historical consumer reads the dropped keys)

## Overview

This release is focused on a single goal: ship a 100% clean dataset bundle for downstream review. A new read-only audit tool exposed 367 latent extraction defects across the 16 datasets; this release closes the most consumer-visible categories (PDF footer leakage, control-character contamination, and bogus `damage:*` identifiers) and reorganizes the `meta.json` manifest around a single per-dataset block.

The remaining `bad_id_format` (hyphenated equipment IDs) and same-named-across-classes `duplicate_id` findings are scoped for v0.25.0 as a separate, breaking ID-contract pass.

## What changed

### Phase Z — `scripts/audit_dataset_quality.py`

A new read-only audit walks every shipped dataset and flags:

- `bad_id_format`, `duplicate_id`
- `control_chars_in_field` (`\t` / `\r` / `\xa0` leaked into `id`/`name`)
- `footer_leakage` (PDF page footer copy left in prose)
- `missing_reference` (broken `*_id` cross-references)
- `inventory_mismatch` (`meta.json.datasets[*].count` vs reality)

```bash
python scripts/audit_dataset_quality.py --output dist/srd_5_1/quality_report.json
```

Severity gating supports CI: `--fail-on critical` exits non-zero on any critical finding.

### Phase A — PDF footer leakage

`postprocess.text.clean_text` + `polish_text` now strip the
`System Reference Document <ver> [<page>]` page footer that the PDF extractor leaves embedded in mid-paragraph text (cross-page spell descriptions, monster action descriptions).

`parse_spells._segment_paragraphs_from_blocks` reruns `clean_text` on the joined paragraph so footer fragments split across blocks are caught after reassembly.

`parse_actions._SAVING_THROW` restricts the captured ability token to actual ability names, so a footer-corrupted action line like `DC 15 System Reference…` no longer mis-parses as `ability:sys` (succubus *Draining Kiss*).

Impact: footer_leakage 56 → 0, succubus DC silently dropped (no canonical recovery possible from corrupted source line).

### Phase B — Feature extraction

`extract_features.py` relaxes the class-feature header filter from `len(text) > 2` to `>= 2` so the monk's two-letter **Ki** feature is no longer dropped.

`parse_features._create_simple_name` now delegates to the canonical `postprocess.ids.normalize_id` helper instead of a local regex that silently swallowed hyphens and left embedded control characters intact. Restores `feature:ki_empowered_strikes` (previously `feature:kiempowered_strikes`) and cleans names like `Giant\t\r\xa0Killer` → `feature:giant_killer`.

Impact: control_chars_in_field 94 → 0, two `feature:*` cross-references resolved.

### Phase C — Damage type identifiers

`parse_monsters._extract_damage_type_id` now scans tokens left-to-right and returns the first canonical D&D 5e damage type word found (acid, bludgeoning, cold, fire, force, lightning, necrotic, piercing, poison, psychic, radiant, slashing, thunder). Returns `None` when no canonical type is present so callers drop the optional `type_id` field rather than emit a bogus identifier.

| Monster | Old `type_id` | New `type_id` |
|---|---|---|
| xorn — "piercing and slashing" | `damage:piercing_and_slashing` | `damage:piercing` |
| archmage — "damage" (from spells) | `damage:damage` | *omitted* |
| archmage — "nonmagical bludgeoning, …" | `damage:nonmagical_bludgeoning` | `damage:bludgeoning` |

Impact: cross-reference validation now passes (3 errors → 0).

### `meta.json` reorganization

The three parallel `files` / `inventory` / `extraction_status` blocks are collapsed into one self-describing per-dataset block:

```json
"datasets": {
  "monsters": {"file": "monsters.json", "count": 317, "status": "complete"},
  "spells":   {"file": "spells.json",   "count": 319, "status": "complete"},
  ...
}
```

**Migration** (every shipping consumer that read these via `meta.json` updates a single key path):

| Old | New |
|---|---|
| `meta["inventory"][ds]` | `meta["datasets"][ds]["count"]` |
| `meta["files"][ds]` | `meta["datasets"][ds]["file"]` |
| `meta["extraction_status"][ds]` | `meta["datasets"][ds]["status"]` |

The three special filenames previously in `meta["files"]` (`meta.json`, `build_report.json`, `index.json`) are well-known and intentionally dropped from the manifest.

## Known issues deferred to v0.25.0

- `bad_id_format`: 126 equipment IDs use hyphens instead of underscores (e.g. `item:battle-axe`). Fixing this is a breaking change to the consumer-facing ID surface — paired with the duplicate-ID disambiguation pass.
- `duplicate_id`: 39 features share names across classes (e.g. `feature:ability_score_improvement` appears in 12 classes). v0.25.0 will namespace these per-class (e.g. `feature:fighter:ability_score_improvement`).

Both are tracked as a single ID-contract pass with migration notes in `docs/INTEGRATION.md`.
