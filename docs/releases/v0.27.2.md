# Release v0.27.2 — `class_targets.py` Retired (P3 Cutover)

**Package Version:** 0.27.2
**Breaking Changes:** none (consumers of the public bundle); see
"Internal API changes" for `parse_classes()` / `parse_features()` signature
changes (only affects callers inside `srd_builder`).

## Overview

v0.27.2 completes P3 of the v0.27.x hand-curated data retirement
campaign begun in v0.27.0. The 763-line
[src/srd_builder/rulesets/srd_5_1/class_targets.py](../../src/srd_builder/rulesets/srd_5_1/class_targets.py)
constant — `CLASS_DATA`, the canonical source of every class's hit_die,
primary_abilities, saving throws, proficiencies, feature lists, subclass
names, page numbers, and 20-level progression — is replaced by a live
five-step PDF extractor.

This is the third and largest hand-curated module retired in the
v0.27.x line:

| File | Lines | Retired |
| --- | --- | --- |
| `lineage_targets.py` | 326 | v0.27.0 P1 |
| `spell_class_targets.py` | 917 | v0.27.0 P2 |
| `class_targets.py` | 763 | **v0.27.2 P3** |
| **Total deleted** | **2,006** | |

The `PROVENANCE.md` registry shrinks from 8 → 5 active hand-curated
entries. The only remaining LIVE-but-disputed entry is
`poison_descriptions.py`, which is extraction-ready but gated by the
damage-formula parser (see v0.27.1 notes).

## What changed

### `extract_classes.py` — five-step pipeline

[src/srd_builder/extract/datasets/extract_classes.py](../../src/srd_builder/extract/datasets/extract_classes.py)
walks pages 8–55 by font fingerprint + bbox geometry:

1. **Discovery** — locate the 12 class headers (24pt GillSans-SemiBold).
2. **Hit die + primary abilities + proficiencies** — Cambria-Bold
   inline-label scan; armor / weapons / tools / skills emitted as the
   same nested dict shape `CLASS_DATA` exposed.
3. **Features + subclasses** — 13.9pt G-SB feature headers; subclass
   roster collected from the class's subclass-table section.
4. **Spellcasting block** — Cambria-Bold labels (`Cantrips`,
   `Spell Slots`, `Spellcasting Ability`, `Spells Known`, `Spell List`)
   parsed into the same nested dict shape full / half / pact casters
   used in `CLASS_DATA`.
5. **Progression** — bbox-aware Features column walk (see below).

### Progression: bbox-aware Features column walk

The progression step is the most involved. Each class's 20-level
progression table lives on the class's first page and is laid out with
the level column on the left and Features as one of the right-hand
columns. Two PDF-layout quirks make this non-trivial:

* **Newspaper layout.** Barbarian's progression is split into two
  visual columns on the same page (L1–L11 at y≈473–682, x≈58;
  L12–L20 at y≈83–247, x≈329). The extractor clusters level cells by x
  and computes one offset per cluster (not per cell — per-cell breaks
  the L1/L10 anchor calculation).
* **False-positive columns.** Warlock has a "Slot Level" column at
  x≈454 whose values match the level regex (1, 2, 3, 4, 5) but are
  duplicates of the actual level column. The extractor's cluster
  validation step drops any cluster whose levels are pure duplicates
  of a larger cluster.

Cell-membership tuning:
* **Strict anchor match** — `-3 <= span.x0 - ix_left <= 30` catches
  continuation spans like the HiraKakuPro apostrophe span of
  `Thieves' Cant` (~0.6pt y jitter, separate span) without bleeding in
  adjacent-column overflow (barbarian L20 had `Unlimited` from the
  Rages column at offset ~64pt — correctly rejected).
* **Y-binning sort** — `round(y / 2.0)` groups spans on the same
  visual line regardless of small y jitter.
* **Hyphen heal** — `clean_text("Ki-\xad‐Empowered")` produces
  `"Ki- Empowered"` (soft hyphen + U+2010 hyphen → space inserted);
  `_PROG_HYPHEN_FIX_RE` heals to `Ki-Empowered`.
* **Apostrophe heal** — separate apostrophe spans sometimes sort
  trailing; `_PROG_APOS_FIX_RE` strips space before any apostrophe
  (`Thieves ' Cant` → `Thieves' Cant`).

### `_PROGRESSION_FIXES` — 4 genuinely-unextractable cells

After the bbox walk, four cells in the SRD progression tables are
genuinely missing letters at the PDF level. Each is filled by
`_PROGRESSION_FIXES` and pinned by a `page.search_for()` reproducer in
[`tests/test_pdf_provenance.py::test_class_progression_truncations_are_real`](../../tests/test_pdf_provenance.py):

| Class | Level | Extracted | Truth |
| --- | --- | --- | --- |
| Barbarian | 11 | `Relentless` | `Relentless Rage` |
| Ranger | 8 | `Land` | `Land's Stride` |
| Rogue | 10 | `Ability Score` | `Ability Score Improvement` |
| Wizard | 20 | `Signature Spell` | `Signature Spells` |

The reproducers assert `page.search_for("Relentless Rage")`,
`page.search_for("Land's Stride")`, etc. return zero hits on their
respective pages — proving the missing text is not recoverable by any
re-application of `get_text()`, only by an override. This satisfies
`AGENTS.md` § *PDF extraction discipline* requirements 1 and 3.

### Snapshot correction: Monk L4

[tests/fixtures/srd_5_1/class_targets_snapshot.json](../../tests/fixtures/srd_5_1/class_targets_snapshot.json)
Monk L4 features were `["Slow Fall", "Ability Score Improvement"]` in
legacy `CLASS_DATA`. The PDF source-of-truth Calibri span at y=344.3
reads `Ability Score Improvement, Slow Fall`. Snapshot updated to
match PDF order (same precedent as P1's `Lightfoot Halfling` →
`Lightfoot` correction).

### Internal API changes

`parse_classes(ruleset)` → `parse_classes(class_data, ruleset)`. The
function now requires the extracted class data instead of importing
the module-level `CLASS_DATA`.

`parse_features(raw_features, source_type, *, ruleset, lineage_data=None)`
gains a `class_data=None` keyword. Required when
`source_type == "class"` (the function now raises `ValueError` if it's
missing). `_build_class_expected_sequence()` takes `class_data` as a
parameter instead of importing the module-level `CLASS_DATA`.

`build.py` calls `extract_classes(pdf_files[0])["classes"]` once inside
the existing `if pdf_files:` block and threads the result into both
`parse_classes()` and `parse_features(source_type="class")`. When no
PDF is present, `parsed_classes` is `[]` (same fallback pattern
lineages/spell_classes use).

## Migration notes for downstream consumers

The published `dist/srd_5_1/classes.json` and `dist/srd_5_1/features.json`
are **unchanged** by this release (same record counts, same IDs, same
fields). Consumers of the bundle need to do nothing.

The only callers affected are internal: anyone who imports
`srd_builder.parse.parse_classes.parse_classes` or
`srd_builder.parse.parse_features.parse_features` directly must update
to the new signatures (see "Internal API changes" above). Nothing
inside this repo imports those functions outside `build.py`, so no
downstream test fixtures or scripts needed updating.

## Closing the v0.27.x line

With P3 shipped, the v0.27.x retirement campaign concludes:

* 3 hand-curated modules retired
* ~2,000 lines of constant data deleted
* `PROVENANCE.md` registry shrunk from 8 → 5 entries
* The "PDF text is corrupted" rationale is now reproducer-disproven
  for every claim it was ever used to justify (lineages, spell lists,
  classes, poisons). Two of those (lineages, spell lists, classes) are
  retired; the fourth (poison_descriptions) is extraction-ready and
  gated only by the damage-formula parser.

The next obvious targets are `equipment_packs.py` /
`equipment_descriptions.py` (LIVE, low payoff per BACKLOG out-of-scope
notes from v0.27.0), and the
`poison_descriptions.py` damage-formula parser work that would unlock
the fifth retirement.
