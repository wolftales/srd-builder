# v0.27.6 — `equipment_descriptions.py` retired (P7 cutover)

## TL;DR

The 398-line hand-curated `*_DESCRIPTIONS` literals in
`src/srd_builder/assemble/equipment_descriptions.py` are replaced by a
live PDF extractor that walks the four equipment description sections
(armor p.63, adventuring gear pp.66–68, tools pp.70–71, lifestyle p.73)
and produces a 69/69 item recovery with two PDF-truth corrections that
the curated literal had silently drifted from.

- **398 lines** of hand-curated Python deleted.
- **+73 new test assertions** (`558 passed`, up from `485`).
- **69-for-69 recovery** across all four sections; 6 description-content
  additions and 2 page-attribution fixes surfaced as drift in the
  retired literal — the new extractor is strictly higher fidelity.
- Closes the v0.27.x retirement line at **~2,856 lines deleted**
  across 6 hand-curated modules.

## Background

The v0.27.5 reproducer (`tests/test_pdf_provenance.py`
`test_equipment_descriptions_section_anchor_extractable`,
`test_equipment_descriptions_item_signature_extractable`, 17 assertions
across 4 sections + 13 item probes — committed in `d942fca`) had
already disproven the `pdf_missing` rationale carried by
`equipment_descriptions.py`: every section's first heading and 13
representative item signatures all recovered cleanly under standard
whitespace normalization.

This release writes the parser, wires it into the build pipeline,
retires the hand-curated literal, and pins the extractor with 73
assertions including a 69-row parametrized snapshot.

## What changed

### NEW `src/srd_builder/extract/datasets/extract_equipment_descriptions.py` (~340 lines)

The parser walks the four sections by **concatenating each section's
pages into one normalized string** and finding heading candidates with a
section-aware regex, then slicing each item's body up to either the
next resolving heading or a known subsection terminator.

**Section table** pins which PDF pages carry which section:

```python
_SECTIONS = [
    ("armor",           [63]),
    ("adventure_gear",  [66, 67, 68]),
    ("tools",           [70, 71]),
    ("lifestyle",       [73]),
]
```

**Heading regex** matches Title-Case words (with optional lowercase
glue words `and`/`or`/`of` in the middle) followed by a period and a
following capital letter:

```python
_HEADING_RE = re.compile(
    r"(?<=[\s.])"
    r"([A-Z][A-Za-z',]+(?:[ ,]+(?:[A-Z][A-Za-z']+|and|or|of))*\.)"
    r"(?= [A-Z])"
)
```

The lowercase-glue alternation was required to recover three items
that an early Title-Case-only regex missed: `block-and-tackle`,
`case-map-or-scroll`, and `potion-of-healing`.

**`_HEADING_TO_ITEM_ID`** is a 69-entry dispatch table mapping each
exact heading string to its stable `item_id`:

```python
_HEADING_TO_ITEM_ID: dict[str, str] = {
    "Padded.":           "item:padded",
    "Acid.":             "item:acid-vial",
    "Disguise Kit.":     "item:disguise-kit",
    "Modest.":           "item:modest",
    "Aristocratic.":     "item:aristocratic",
    …
}
```

The `Wretched.` lifestyle heading is deliberately omitted because it
has no cost row in the lifestyle table and therefore no
`item_id` to attach to.

**Subsection terminators** stop body slicing at sub-headings inside
the same section concatenated text:

```python
_SUBSECTION_TERMINATORS: dict[str, tuple[str, ...]] = {
    "armor":          ("Heavy Armor ", "Medium Armor ", "Armor Armor Cost"),
    "adventure_gear": ("Adventuring Gear",),
    "tools":          ("Mounts and Vehicles",),
    "lifestyle":      ("Self-Sufficiency",),
}
```

These were necessary because the SRD lays subsection sub-headings
(`Heavy Armor`, `Mounts and Vehicles`, `Self-Sufficiency`, …)
mid-section as table-row sentinels rather than page breaks. Without
them, `half-plate` would bleed into "Heavy Armor", `aristocratic`
into "Self-Sufficiency", and so on.

**Normalization handles four PDF artifacts** that the legacy literal
had each handled with a different ad-hoc patch (or just stripped the
content — see drift catalogue below):

```python
_SOFT_HYPHEN_RE = re.compile(r"[-\xad\u2010\u2011]{2,}")  # 1 → soft hyphens collapse
                                                          # 2 → curly U+2019 → ASCII '
_PAGE_FOOTER_RE = re.compile(r"\s*System Reference Document 5\.1 \d+\s*")
                                                          # 3 → strip running footer
_EM_DASH_SPACE_RE = re.compile(r" *— *")                  # 4 → close em-dash line wraps
```

```python
            "name":        "Padded",
            "page":        63,

```

### MODIFIED `src/srd_builder/assemble/assemble_equipment.py`

`assemble_equipment_from_tables` gained an `equipment_descriptions=`
kwarg, mirroring the v0.27.5 `equipment_packs=` shape:

```python
def assemble_equipment_from_tables(
    tables: dict[str, Any],
    ruleset: str,
    *,
    equipment_packs: list[dict[str, Any]] | None = None,
    equipment_descriptions: list[dict[str, Any]] | None = None,
) -> list[dict[str, Any]]:
    …
    _add_item_descriptions(equipment_items, equipment_descriptions)
```

`_add_item_descriptions` no longer imports `get_description_lookup`
from the retired `equipment_descriptions` module. It builds the lookup
inline from the parameter list:

```python
def _add_item_descriptions(items, descriptions):
    if descriptions is None:
        return
    lookup = {entry["item_id"]: entry["description"] for entry in descriptions}
    for item in items:
        if (desc := lookup.get(item["item_id"])) is not None:
            item["description"] = desc
```

### MODIFIED `src/srd_builder/build.py`

A new helper `_extract_equipment_pdf_data` consolidates both equipment
PDF extractors so the `build()` orchestrator stays under the
`PLR0915` 160-statement ceiling:

```python
from .extract.datasets.extract_equipment_descriptions import extract_equipment_descriptions

def _extract_equipment_pdf_data(
    ruleset: str,
) -> tuple[list[dict[str, Any]] | None, list[dict[str, Any]] | None]:
    ruleset_dir = Path("rulesets") / ruleset
    pdf_files = sorted(ruleset_dir.glob("*.pdf"))
    if not pdf_files:
        return None, None
    pdf_path = pdf_files[0]
    packs        = extract_equipment_packs(pdf_path)["packs"]
    descriptions = extract_equipment_descriptions(pdf_path)["descriptions"]
    return packs, descriptions
```

The equipment-assembly block in `build()`:

```python
if parsed_tables:
    equipment_packs, equipment_descriptions = _extract_equipment_pdf_data(ruleset)
    parsed_equipment = assemble_equipment_from_tables(
        parsed_tables,
        ruleset,
        equipment_packs=equipment_packs,
        equipment_descriptions=equipment_descriptions,
    )
```

## Drift caught vs. retired literal

Pre-deletion direct comparison between the new extractor and the
retired `*_DESCRIPTIONS` lists surfaced **eight discrepancies** that
all favour the live PDF as ground truth:

**Two `page` corrections** — the curated literal said p. 67 for these
two adventure-gear items; the PDF places both on p. 66:

| `item_id`            | curated `page` | actual PDF page |
| -------------------- | --------------:| ---------------:|
| `item:antitoxin-vial`| 67             | 66              |
| `item:arcane-focus`  | 67             | 66              |

**Six description-content additions** — the curated literal had
editorially stripped parenthetical cross-references and table mentions
that the SRD prose carries inline. The new extractor restores them:

| `item_id`              | restored phrase |
| ---------------------- | --------------- |
| `item:book`            | `(described later in this section)` |
| `item:gaming-set`      | `Tools table examples` reference  |
| `item:holy-symbol`     | `Appendix PH-B` cross-reference   |
| `item:musical-instrument` | `Tools table` reference         |
| `item:pouch`           | `(described earlier in this section)` |
| `item:artisans-tools`  | `table shows examples` reference  |

These are not bugs to suppress — they are the SRD's own text. The
curated literal had been silently editing them out for unknown reasons.
The new extractor reflects the PDF as written, and the per-item
snapshot test pins each description to its current PDF-truth value.

## What did NOT change

- **No fixture regeneration.** Equipment fixtures already aligned with
  the build pipeline; the descriptions are item-level fields the new
  extractor populates the same way the retired literal did.
- **No schema changes.** `equipment.json` still carries `description`
  on adventure-gear, tools, armor, and lifestyle records.
- **No bump to `extract_equipment.py` page-constants.** The
  descriptions extractor uses its own per-section `_SECTIONS` table;
  the existing parametrized cross-extractor audit (v0.27.4) covers the
  four `EQUIPMENT_*_PAGE` / `SPELL_*_PAGE` / etc. constants and is
  unaffected.
