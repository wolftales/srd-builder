# Release v0.26.0 — Generated Schema Exemplars + Provenance Registry

**Release Date:** June 17, 2026
**Package Version:** 0.26.0
**Breaking Changes:** None
**Migration:** None

## Overview

v0.26.0 is a cleanup, instrumentation, and discovery release. It
finishes the v0.25.0 "make the bundle self-describing" thread by adding
generated schema exemplars to the bundle, formalizes the registry of
hand-curated data sources via `docs/PROVENANCE.md`, and adds the first
reproducer-style test framework that pins our assumptions about the
source PDF.

The most interesting outcome is unplanned: the lineage reproducer test
**disproved** a long-standing "PDF text is corrupted" claim and opens
the door to retiring three of the eight hand-curated overrides in
v0.27.0.

## What changed

### Generated schema exemplars (replaces `docs/templates/`)

`scripts/generate_exemplars.py` now derives a minimal valid instance
from each of the 16 dataset schemas in `schemas/`. The outputs land in
`schemas/exemplars/*.exemplar.json`, are committed to the repo, and
ship in every `make bundle` under `<bundle>/schemas/exemplars/` —
co-located with the schemas they instantiate.

The previous `docs/templates/` directory only covered 11 of 16 schemas
and drifted whenever a schema version bumped. The generator is driven
by the schema itself, so the exemplars stay in lockstep automatically.

- `tests/test_schema_exemplars.py` runs the generator in `--check` mode
  on every test run, failing CI if any schema can no longer produce a
  valid instance.
- `docs/templates/TEMPLATE_*.json` (5 files) are deleted.
- `test_template_files_have_version_note` is removed.

### `docs/PROVENANCE.md` — registry of hand-curated sources

A new registry documents every place in `src/` where a hand-typed table
or override replaces direct PDF extraction. Each entry carries a
`reason_code` (one of `pdf_corruption`, `pdf_missing`,
`cross_reference_glue`, `derived_lookup_table`, `alias`), the affected
PDF pages, a downstream dataset list, and (when written) a pointer to a
reproducer test.

Eight sources are registered: `class_targets`, `lineage_targets`,
`spell_class_targets`, `poison_descriptions_manual`,
`equipment_extended`, `equipment_packs`, `equipment_descriptions`, and
the `extract_equipment` page constants.

### Reproducer test framework (`tests/test_pdf_provenance.py`)

The new test module pins the *current* extraction behavior of disputed
PDF regions. A reproducer is not a regression guard — when it fails,
our model of the source PDF has shifted and the matching manual
override is up for re-evaluation.

Tests skip cleanly when the SRD PDF is not present (CI / container
builds without rulebooks).

### Key finding: lineage "corruption" claim is FALSE

`src/srd_builder/srd_5_1/lineage_targets.py` has carried this
docstring for years: *"PDF text is corrupted; manually transcribed via
visual inspection."* The first reproducer was written to confirm that
claim. It failed.

Under `pymupdf 1.27.x`, the lineage pages (PDF indices 2–6) are fully
extractable once whitespace is normalized — the SRD body text uses
tab + carriage-return + non-breaking-space sequences as word
separators rather than ordinary spaces. After
`re.sub(r"[\t\r\xa0]+", " ", text)`, all 10 tested lineage keywords
(Dwarf, Hill Dwarf, High Elf, Halfling, Lightfoot, Dragonborn, Gnome,
Half-Elf, Half-Orc, Tiefling) appear on their expected pages.

The implication ripples: `spell_class_targets.py` ("PDF text is
corrupted; manually mapped via visual inspection" for pp. 105–113) and
the implicit corruption rationale on `class_targets.py` are the same
author / same era / same vague justification. Treating those as
retire-able in v0.27.0 is a strong bet.

### Dead code removed

`src/srd_builder/extraction/reference_data.py` (734 lines) is deleted:

- `CALCULATED_TABLES` — empty
- `PRICING_TABLES` — empty
- `REFERENCE_TABLES` — single entry `spell_slots_by_level`, already
  superseded by per-class progression data
- `VALIDATION_CLASS_PROGRESSIONS` — duplicate of
  `tests/validation_data.py` with different formatting
- `get_table_data()` — zero importers in `src/srd_builder/`

The one useful payload, `TEXT_PARSED_TABLES` (16 table-page anchors),
is promoted to `tests/fixtures/expected_table_pages.json` as a fixture
for a future TOC verifier. Net repo shrinks **−625 lines** (734
deleted, 109 fixture).

## What's next (v0.27.0)

- Retire `LINEAGE_DATA` by writing a real lineage parser against the
  whitespace-normalized PDF text.
- Apply the same reproducer probe to `spell_class_targets.py` and
  `class_targets.py`. If either claim of "corruption" also fails to
  reproduce, write parsers for those as well.
- Goal: shrink the PROVENANCE registry from 8 to 4–5 by displacing the
  three corruption-rationale entries with real extraction.
